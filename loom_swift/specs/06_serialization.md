## 1. Purpose

This document specifies the XML serialization layer — the formats, loaders, and conventions used to persist and load every category of Loom data. It covers:

- The full XML schema for each file type
- Each loader: approach, quirks, and data classes produced
- The Bezier polygon file format (external format, generated by the editor)
- The lenient parsing strategy and the DTD workaround
- What can and cannot currently be serialized back to XML
- Design assessment and improvement recommendations

|------|--------|----------|-------------|
| `configuration/global_config.xml` | `GlobalConfigLoader` | `GlobalConfig` | Yes |
| `configuration/shapes.xml` | `ShapeConfigLoader` | `ShapeLibrary` | No |

All loaders use `scala.xml` (Scala's built-in XML library). Most use a **lenient SAX configuration** that disables DTD resolution (see §3). `GlobalConfigLoader` is the only loader with a corresponding save method.

## 3. Lenient Parsing Strategy

### 3.1 The Problem

Polygon files exported by the Bezier editor include an ISO-8859-1 DOCTYPE declaration referencing an external DTD:

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
<!DOCTYPE regularPolygon SYSTEM "../../dtd/regularPolygon.dtd">
```

The default `scala.xml.XML.loadFile()` passes this to the JVM's SAX parser which attempts to fetch the DTD. The DTD path is relative, so it resolves to nothing and the parse fails.

### 3.2 The Solution

All loaders that may encounter Bezier-generated files use a custom SAX factory configuration:

```scala
private def loadXMLLenient(filePath: String): Elem = {
  val factory = javax.xml.parsers.SAXParserFactory.newInstance()
  factory.setValidating(false)
  factory.setFeature(
    "http://apache.org/xml/features/nonvalidating/load-dtd-grammar", false)
  factory.setFeature(
    "http://apache.org/xml/features/nonvalidating/load-external-dtd", false)
  val saxParser = factory.newSAXParser()
  XML.withSAXParser(saxParser).loadFile(filePath)
}
```

This suppresses DTD fetching and validation entirely while still parsing the document. All polygon, open curve, point, and oval loaders use this helper.

### 3.3 Safe Access Helpers

All modern loaders avoid direct `.text.toInt` calls. Instead they use a family of helpers that return defaults on any failure:

```scala
getTextOrDefault(node, elem, default: String): String
getIntOrDefault(node, elem, default: Int): Int
getBoolOrDefault(node, elem, default: Boolean): Boolean
getColorOrDefault(node, elem, default: Color): Color  // reads r/g/b/a attributes
getDoubleOrDefault(node, elem, default: Double): Double
```

The result is that a missing or malformed element silently uses the declared default. No validation failure is reported to the caller.

| `INLINE_POINTS` | 2 | Point list embedded directly in `shapes.xml` |
| `OPEN_CURVE_SET` | 3 | References a named set in `curves.xml` |
| `POINT_SET` | 4 | References a named set in `points.xml` |
| `OVAL_SET` | 5 | References a named set in `ovals.xml` |

### 5.3 Shape3D Type Constants

| `type` attribute | Value | Description |
|-----------------|-------|-------------|
| `NONE` | 0 | 2D only |
| `CRYSTAL` | 1 | 3D faceted polyhedron |
| `RECT_PRISM` | 2 | Rectangular box extrusion |
| `EXTRUSION` | 3 | Linear extrusion |
| `GRID_PLANE` | 4 | 2D grid |
| `GRID_BLOCK` | 5 | 3D grid |

## 6. polygons.xml and Polygon Files

**Loaders:** `PolygonConfigLoader` (for `polygons.xml`), then recursively loads individual polygon XML files.

### 6.1 polygons.xml Schema

```xml
<PolygonSetCollection>
  <PolygonSet name="string">

    <!-- Option A: load from file -->
    <Source type="file">
      <File>filename.xml</File>           <!-- relative to resources/polygonSet/ -->
      <PolygonType>SPLINE|LINE</PolygonType>
      <Filter>all|closed_only</Filter>
    </Source>

    <!-- Option B: computed regular/star polygon -->
    <Source type="regular">
      <TotalPoints>int</TotalPoints>
      <InternalRadius>double</InternalRadius>
      <Offset>double</Offset>
      <ScaleX>double</ScaleX>  <ScaleY>double</ScaleY>
      <RotationAngle>double</RotationAngle>
      <TransX>double</TransX>  <TransY>double</TransY>
      <PositiveSynch>boolean</PositiveSynch>
      <SynchMultiplier>double</SynchMultiplier>
    </Source>

  </PolygonSet>
</PolygonSetCollection>
```

### 6.2 Individual Polygon XML File Schema (Bezier Editor Format)

Individual polygon files are generated by the Bezier editor. They use ISO-8859-1 encoding and an unresolvable DOCTYPE (hence the lenient parser).

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
<!DOCTYPE regularPolygon SYSTEM "../../dtd/regularPolygon.dtd">

<polygonSet>
  <polygon isClosed="true|false">
    <curve>
      <point x="double" y="double" pressure="double"/>  <!-- P0: anchor -->
      <point x="double" y="double"/>                    <!-- P1: control out -->
      <point x="double" y="double"/>                    <!-- P2: control in -->
      <point x="double" y="double"/>                    <!-- P3: anchor -->
    </curve>
    <!-- more <curve> elements — each is one cubic Bézier segment -->
  </polygon>
  <!-- more <polygon> elements -->

  <scaleX>double</scaleX>
  <scaleY>double</scaleY>
  <rotationAngle>double</rotationAngle>
  <transX>double</transX>
  <transY>double</transY>
</polygonSet>
```

**Point encoding:** Each `<curve>` contains exactly 4 points representing one cubic Bézier segment: anchor, control-out, control-in, anchor. Consecutive curves share anchor points: P3 of curve N is identical to P0 of curve N+1.

**Coordinate system:** All coordinates are in normalised space (approximately −1.0 to +1.0 with origin at centre). The `transX`/`transY` fields at the end of the file encode a translation from the original editor layout.

**`pressure` attribute:** Present on P0 (anchor) points only. Used by brush stamping for pressure-sensitive size/opacity scaling. Omitted if pressure data is unavailable; defaults to 1.0.

**`isClosed` attribute:** `true` → `SPLINE_POLYGON`; `false` → `OPEN_SPLINE_POLYGON`.

### 6.3 Polygon Types Produced

| Loader method | Condition | PolygonType produced |
|--------------|-----------|---------------------|
| `loadSplinePolygonsFromFile()` | `isClosed="true"` | `SPLINE_POLYGON` |
| `loadSplinePolygonsFromFile()` | `isClosed="false"` | `OPEN_SPLINE_POLYGON` |
| `loadLinePolygonsFromFile()` | *(any)* | `LINE_POLYGON` |

`loadLinePolygonsFromFile()` reads only points at indices 0 and 3 of each curve (the two anchors), discarding the control points entirely. This produces simpler straight-edged polygons from the same source file.

### 6.4 PolygonSetWriter

`PolygonSetWriter` can write a `PolygonSet` back to the Bezier polygon XML format. This is the only non-`GlobalConfig` write path currently implemented. It is used by the subdivision baking pipeline to persist computed polygon sets.

## 7. curves.xml, points.xml, ovals.xml

These three files follow the same structural pattern as `polygons.xml` — a collection element containing named set elements, each referencing an external file.

### 7.1 Open Curve Files

```xml
<openCurveSet>
  <openCurve>
    <curve>
      <point x="double" y="double" pressure="double"/>
      <!-- 4 points per curve, same Bézier encoding as polygon files -->
    </curve>
    <!-- more <curve> elements -->
  </openCurve>
  <!-- more <openCurve> elements -->
</openCurveSet>
```

Produces `OPEN_SPLINE_POLYGON`. Always open — no `isClosed` attribute.

### 7.2 Point Files

```xml
<pointSet>
  <point x="double" y="double" pressure="double"/>
  <!-- each <point> becomes one POINT_POLYGON with a single vertex -->
</pointSet>
```

### 7.3 Oval Files

```xml
<ovalSet>
  <oval cx="double" cy="double" rx="double" ry="double"/>
  <!-- each oval stored as OVAL_POLYGON(Vector2D(cx,cy), Vector2D(cx+rx, cy+ry)) -->
</ovalSet>
```

The two-point encoding (`centre` + `radii endpoint`) is an internal convention. At render time `Sprite2D.drawOval()` reconstructs the ellipse from these two points.

  <!-- Transform plugin configuration (5 plugins) -->
  <TransformSet>

    <ExteriorAnchors enabled="boolean">
      <Probability>double</Probability>
      <SpikeFactor>double</SpikeFactor>
      <WhichSpike>ALL | CORNERS | MIDDLES</WhichSpike>
      <SpikeType>SYMMETRICAL | RIGHT | LEFT | RANDOM</SpikeType>
      <SpikeAxis>XY | X | Y</SpikeAxis>
      <RandomSpike>boolean</RandomSpike>
      <RandomSpikeFactor min="double" max="double"/>
      <CpsFollow>boolean</CpsFollow>
      <CpsFollowMultiplier>double</CpsFollowMultiplier>
      <CpsSqueeze>boolean</CpsSqueeze>
      <CpsSqueezeFacto>double</CpsSqueezeFacto>   <!-- note: typo in source -->
    </ExteriorAnchors>

      <TearFactor>double</TearFactor>

### 8.3 Known Typo

The XML element `<CpsSqueezeFacto>` (missing final `r`) is used throughout both the loader and all existing XML files. It must remain as-is for compatibility; do not correct the spelling in Scala without also updating every XML file.

The field `polysTranformWhole` (missing `s` in `Transform`) is similarly present in both loader and data class. See spec §02 for full details.

### 8.4 SubdivisionParams Data Class (key fields)

```scala
class SubdivisionParams(val name: String) {
  var subdivisionType: Int
  var visibilityRule: Int
  var ranMiddle: Boolean
  var ranDiv: Double
  var lineRatios: Vector2D          // default (0.5, 0.5)
  var controlPointRatios: Vector2D  // default (0.25, 0.75)
  var insetTransform: Transform2D
  var continuous: Boolean
  var polysTransform: Boolean
  var polysTransformPoints: Boolean
  var polysTranformWhole: Boolean   // typo preserved
  // 15+ PTW_ fields
  var transformSet: Array[Transform]  // 7 slots, indices match plugin type constants
}
```

### 9.3 Color Attribute Format in Renderer Element

The `strokeColor` and `fillColor` attributes on `<Renderer>` use space-separated `key=value` pairs rather than separate XML attributes:

```
strokeColor="r=0 g=128 b=255 a=200"
```

This is parsed by splitting on spaces and then on `=`. It differs from the RGBA element format used elsewhere (`<Color r="0" g="0" b="0" a="255"/>`). This inconsistency is a serialization fragility point.

        </RandomScale>

## 12. Design Assessment

### S1 — No schema validation

There are no XSD (XML Schema Definition) or Relax NG files for any format. This means:
- Malformed documents are silently accepted with default values substituted.
- No tool can validate an XML file before loading it.
- The spec you are reading (this document) is the only authoritative schema description.

**Swift:** Consider using `Codable` with explicit decoding errors, or define XSD files for tooling validation.

---

### S2 — Two incompatible color formats

Colors appear in three different forms across the codebase:

| Context | Format | Example |
|---------|--------|---------|
| Legacy `Config.borderColor` | Comma-separated RGB string | `"0,0,0"` |
| Modern XML elements | Four RGBA attributes | `<Color r="0" g="0" b="0" a="255"/>` |
| `<Renderer>` attributes | Space-separated key=value | `strokeColor="r=0 g=0 b=0 a=255"` |

All three are parsed by different code paths. Any serialization library will need to handle all three.

---

### S3 — Preserved typos create a compatibility contract

Two typos in field names and XML element names have become part of the de-facto schema:

- `<CpsSqueezeFacto>` (ExteriorAnchors, CentralAnchors, AnchorsLinkedToCentre)
- `polysTranformWhole` field name in `SubdivisionParams`

These cannot be corrected in Scala without simultaneously updating every XML file that uses them. The Swift rewrite can and should use correct spelling, but must be able to read both spellings during any migration period.

---

### S4 — No round-trip for subdivision, rendering, or sprite configuration

The most complex and most user-editable parts of the system (subdivision parameters, rendering configuration) have no write path. There is no way to save a modified configuration back to disk from the running application. The `loom_parameter_editor` Python application writes these files by generating XML from scratch rather than modifying the Scala structures.

**This is the primary serialization gap to address before a GUI editor for the Swift version is possible.**

---

### S5 — Pressure data is inconsistently present

The `pressure` attribute on polygon points is optional and present only on P0 (anchor) points. Loaders silently default to 1.0 when absent. Any new polygon format should document whether pressure data is present and make it consistent (either always present or stored separately).

---

### S6 — Path resolution is ad-hoc

Each loader constructs resource file paths differently:
- Some use string concatenation with `File.separator`.
- Some use `ProjectFilePath` utilities.
- Some accept a `resourcesBasePath` parameter; others hard-code the path convention.

This makes the loading pipeline fragile when project directories move, and there is no central path resolution authority.

## 13. Verified Corrections (from Test_051)

Reading an actual project (`Test_051`) against a live `.loom_projects` directory revealed the following corrections to earlier sections of this spec. These are authoritative — use these over any conflicting description above.

### Directory structure
Config files live in `configuration/`, not the project root. Polygon files live in `polygonSets/`, not `resources/polygonSet/`. See §2.1 for the corrected layout.

### Polygon file path resolution
`polygons.xml` uses `<Folder>` and `<Filename>` child elements within `<Source type="file">`, not a single path element:
```xml
<Source type="file">
  <Folder>polygonSet</Folder>
  <Filename>goodcat_mesh_mesh.xml</Filename>
  <PolygonType>SPLINE_POLYGON</PolygonType>
</Source>
```
`<Folder>` maps to `polygonSets/` in the project root regardless of its value (the folder name in the XML may not match exactly — resolve relative to the project root `polygonSets/` directory).

### Polygon files in practice
The Test_051 polygon files have no DOCTYPE declaration and use UTF-8 encoding. The lenient SAX parser is still required for safety (other projects may differ) but is not exercised by this project.

### InsetTransform Rotation attribute
The actual format uses `x`/`y` attributes, not `angle`:
```xml
<Rotation x="0.0" y="0.0"/>
```
The Swift loader should read the `x` attribute for the rotation value.

### TransformSet element
`<TransformSet>` is absent entirely when no transform plugins are configured. The loader must treat its absence as "all plugins disabled" rather than an error.

### EditorExtensions in sprites.xml
The parameter editor writes an `<EditorExtensions>` block inside each `<Sprite>`. The engine loader should ignore it silently:
```xml
<EditorExtensions enabled="true" animator="random">
  <RotationOffset .../>
  <ScaleFactor .../>
  <!-- etc -->
  <Geometry .../>   ← redundant shape/subdivision info; ignore
</EditorExtensions>
```

### Renderer set name "DefaultSet"
Sprites written by the parameter editor may reference `<RendererSet name="DefaultSet"/>` even when no renderer set with that name exists in `rendering.xml`. The Scala engine falls back to the first available renderer set. The Swift `SceneAssembler` should preserve this: if the named renderer set is not found, use the first renderer set in the library rather than throwing an error.

### Rendering XML: Renderer element format
The actual `rendering.xml` uses child elements for renderer properties, not attributes:
```xml
<Renderer name="string">
  <Mode>STROKED</Mode>
  <StrokeWidth>1.0</StrokeWidth>
  <StrokeColor r="0" g="0" b="0" a="255"/>
  <FillColor r="0" g="0" b="0" a="255"/>
  <PointSize>2.0</PointSize>
  <HoldLength>1</HoldLength>
  <PointStyle stroked="true" filled="true"/>
</Renderer>
```
The earlier schema in §9.2 showed attributes on the `<Renderer>` element — that reflects the Scala source code's setter calls, not the actual XML on disk. The child-element form is what the loader must parse.

## 14. Swift Implementation Status

The Swift serialization layer is fully implemented. Key decisions made:

| Concern | Swift solution |
|---------|---------------|
| XML parsing | `XMLParser` (SAX) via `XMLNode` helper in `Loaders/XMLNode.swift` |
| DTD problem | `XMLConfigLoader` disables DTD resolution; `XMLPolygonLoader` strips DOCTYPE via regex before parsing |
| Color format | `LoomColor` struct handles all three input formats; stored internally as RGBA Double |
| Schema validation | Safe defaults throughout; missing fields silently use struct defaults |
| Save-back | `JSONConfigLoader.save(_:to:)` writes full `ProjectConfig` as JSON — complete roundtrip |
| Typo compatibility | `XMLConfigLoader` reads `CpsSqueezeFacto` and `polysTranformWhole` as written; JSON uses correct spelling |
