# Loom Integration

bezier_py is launched as a subprocess by the Loom Parameter Editor (LPE) and produces XML files that the Scala Loom engine consumes. This document covers the full integration contract between the three components.

---

## Launch Protocol

The LPE's `SpriteTab` launches bezier_py via `QProcess` when the user edits a shape. The command template:

```
python <BEZIER_PY_PATH>/main.py \
    --save-dir <project>/polygonSets \
    --load <project>/polygonSets/<name>.poly.xml \
    --name <name>
```

Constants in `sprite_tab.py`:
```python
BEZIER_PY = "/Users/broganbunt/Loom_2026/bezier_py"
PYTHON    = "python3"
```

The `--save-dir` argument is always the project's `polygonSets/` directory. bezier_py creates this directory if it does not exist.

---

## File Naming Convention

The LPE expects saved files in `polygonSets/` using the naming convention:

| File | Description |
|---|---|
| `<name>.poly.xml` | Single-layer closed polygon set |
| `<name>.curve.xml` | Single-layer open curve set |
| `<name>.layers.xml` | Multi-layer manifest |
| `<name>_<layer>.xml` | Per-layer polygon files (multi-layer) |

The LPE's `PolygonConfigIO` indexes files by the basename without extension, so `myshape.poly.xml` is registered as `myshape`.

---

## DOCTYPE Protocol

### Why DOCTYPE exists

The Scala engine's XML loader (`scala-xml`) requires a DOCTYPE declaration to recognise the file as a valid `polygonSet`. Without it, the engine silently rejects or misparses the file.

### Who writes DOCTYPE

bezier_py writes `<!DOCTYPE polygonSet SYSTEM "polygonSet.dtd">` as the second line of every `polygonSet` file:

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
<!DOCTYPE polygonSet SYSTEM "polygonSet.dtd">
<polygonSet>
```

### LPE's role

The LPE reads polygon XML files produced by bezier_py. When it needs to pass a file to the Scala engine, it ensures the DOCTYPE is present. The legacy `ShapeConfigIO` reading path strips DOCTYPE before ET parsing (same `_strip_doctype` pattern).

### Load-time stripping

`polygon_set_xml.read_polygon_set` strips DOCTYPE via regex before passing to `ElementTree.fromstring`, because Python's `xml.etree` does not handle DOCTYPE declarations.

---

## Output After Save

On save bezier_py:
1. Writes the polygon/oval/point/layer XML to `save_dir`.
2. Auto-exports one or more `.svg` files alongside the XML.

The LPE does not use the SVG files; they are for external use.

---

## Auto-Save on Window Close

When the bezier_py window is closed (`closeEvent`):
- If any geometry (polygons, ovals, or discrete points) exists: auto-save using the same dispatch as the manual Save action.
- Save destination: `--save-dir` + `--name` from the CLI arguments.
- If no geometry exists: close without saving (does not overwrite an existing file with an empty set).

This ensures that the LPE's subprocess always produces a saved file when the user closes the window after drawing.

---

## Coordinate Contract

The Scala engine reads normalised coordinates in the range `−0.5` to `+0.5` (approximately). The exact pipeline bezier_py uses:

```
XML value = round(round((canvas_px / 1000 − 0.5 − 0.02) × 100)) / 100
```

The engine's reader applies the inverse:
```
canvas_px = (xml_val + 0.02) × 1000 + 500
```

This is fully documented in `bezier_py/TECHSPEC.md` and must be preserved exactly in the Swift reimplementation.

---

## Multi-layer Files and the Engine

The Scala engine does not understand `.layers.xml` manifests. When bezier_py saves a multi-layer file, each per-layer polygon XML is a standalone `polygonSet` file that the engine can read independently. The LPE registers each per-layer file as a separate polygon set entry.

---

## LPE ↔ bezier_py Data Flow Summary

```
LPE (SpriteTab)
    → launches: python bezier_py/main.py --save-dir polygonSets/ --load <file> --name <name>
    ← produces: polygonSets/<name>.poly.xml  (+ optional .layers.xml, per-layer .xml, .svg)

LPE (PolygonConfigIO)
    → reads:  polygonSets/<name>.poly.xml
    → updates PolygonSetLibrary

Scala engine
    → reads:  polygonSets/<name>.poly.xml (via shapes.xml auto-generated by LPE)
    → renders subdivided geometry
```
