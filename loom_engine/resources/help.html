Loom Engine — Help

Loom is a Scala-based drawing engine that generates complex visual patterns from geometric descriptions defined in the Loom Parameter Editor. It reads a set of XML configuration files, constructs a scene from polygon sets, open curve sets, and point sets, and renders it to a Swing window at any resolution.

What Loom does

  • Loads geometry (polygons, open curves, points) from XML files created in Bezier Draw
  • Recursively subdivides closed polygon shapes through configurable multi-stage subdivision pipelines
  • Applies point-level geometric transforms (spiking, curving, tearing) at each subdivision stage
  • Renders the resulting geometry using stroked outlines, solid fills, point stamps, or brush-texture stamps
  • Animates sprites with random jitter, keyframe interpolation, or polygon-vertex morph targets
  • Exports high-resolution stills and animation frame sequences as PNG files
Bezier Draw → XML geometry files → Parameter Editor → XML config files
XML config files → Loom Engine → Canvas / PNG output

Starting Loom

From the Parameter Editor (normal workflow)

Click Run Loom (Cmd+L) in the Parameter Editor's Run tab. The editor auto-saves all config files, then launches sbt run from the Loom engine directory. SBT compiles the Scala sources (takes 10–30 seconds on first run; subsequent runs are faster) and opens two windows:

  1. Project Selector — a small dialog to choose which project to load.
  2. Loom Canvas — the main rendering window.

Project Selector dialog

On startup, a Project Selector dialog appears:

The Parameter Editor's Run Loom button passes the current project's directory to the engine automatically — the Project Selector is pre-populated with the correct project. You typically just click Load Project without changing anything.

Direct launch (advanced)

You can also launch from a terminal:

cd /Users/broganbunt/Loom_2026/loom_engine
SBT_OPTS="-Xmx8G" sbt run

The Project Selector dialog appears and you choose the project manually.

The Loom Window

The main Loom canvas is a Swing window showing the rendered output. Its pixel dimensions are Canvas Width × Quality Multiple by Canvas Height × Quality Multiple as configured in the Parameter Editor's Global Config tab. When the quality multiple is greater than 1, the window is scaled down to fit on screen; the full-resolution image is still composited internally and saved to PNG at full size.

Window behaviour

  • Static mode (Animating off): one frame is drawn and held. The window does not update until a reload.
  • Animating mode (Animating on): the draw loop runs continuously. Draw cycle counts are printed to the console every 100 frames.
  • Paused: animation loop is suspended. The current frame remains displayed.
For fine detailed work, leave Animating off, adjust parameters in the editor, and press F5 to reload. This lets you evaluate each change as a static frame.

Rendering Pipeline

Each time Loom initialises (on startup or reload), it executes the following pipeline:

  1. Load config — reads global_config.xml, polygons.xml, curves.xml, points.xml, subdivision.xml, shapes.xml, sprites.xml, rendering.xml.
  2. Load geometry — parses polygon set XML files from polygonSets/, open curve set XML files from curveSets/, and point set XML files from pointSets/.
  3. Build shapes — for each Shape definition, pairs the geometry source with its subdivision parameters.
  4. Subdivide (polygon types only) — recursively applies each SubdivisionParams stage to every polygon, producing the final subdivided mesh. Open curves and point sets bypass this step.
  5. Build sprites — instantiates each Sprite with its shape, position, size, and animation configuration.
  6. Draw loop — each frame: clear/overlay background, then for each sprite: apply transforms (animation, morph), then for each polygon/curve/point: apply renderer and draw.
Subdivision is computed once at initialisation, not per-frame. Very large subdivision counts (many stages or many sprites) may slow the initial load but have no effect on animation frame rate.

Polygon types: SPLINE vs LINE

Polygon sets can use two polygon types, selected per-set in the Parameter Editor:

  • SPLINE_POLYGON — bezier control points are active. Subdivision uses curved edges. Use with spline subdivision modes.
  • LINE_POLYGON — only anchor points are used; all edges are straight. Produces faceted, angular geometry. Use with LINE subdivision modes.

Open curves

Open curves are non-closed cubic Bézier paths with a defined start and end. They are drawn in Bezier Draw using the Finish Open Curve button. Loom renders each open curve as a continuous stroked path or brush-stamp chain. They participate in sprite transforms (position, scale, rotation, jitter animation, keyframe animation) but are not subdivided and do not support morph targets.

Point sets

Point sets are collections of discrete anchor positions. Loom renders each point as a dot stamp (POINTS mode) or brush stamp (BRUSHED mode). They participate in sprite transforms and jitter/keyframe animation but are not subdivided and do not support morph targets.

Subdivision

Subdivision is Loom's core pattern-generation mechanism. It applies only to polygon-type shapes (polygon sets and regular polygons). Open curves and point sets are not subdivided.

How it works

A Params Set contains one or more SubdivisionParams stages. Each stage is applied recursively to the output of the previous stage:

  • Stage 1 divides each base polygon into sub-polygons.
  • Stage 2 divides each of those into more sub-polygons.
  • … and so on.

The total polygon count grows multiplicatively. A square with 3 QUAD stages → 4³ = 64 polygons. A square with 5 QUAD stages → 1024 polygons.

Point-level transforms

Each SubdivisionParams stage can apply transforms to specific subsets of points within each sub-polygon:

  • Inner control points — adjust curvature of internal subdivision edges.
  • Outer control points — puff, pinch, or S-curve the outer edges (PUFF, PINCH, PUFF_PINCH, etc.).
  • Anchors linked to centre — tear outer anchors toward corners or the centre, creating fan/burst patterns.
  • Inset transform — scale/rotate the whole sub-polygon within its parent, creating gaps or overlaps.
  • Whole polygon transforms (PTW) — randomly translate/scale/rotate selected sub-polygons as units.

Renderer Modes

Each renderer set contains one or more named renderers. The set's playback mode (STATIC, SEQUENTIAL, or RANDOM) determines which renderer is used for each polygon.

Dynamic changes

Stroke width, stroke colour, fill colour, and point size can all be animated dynamically per-polygon or per-frame using configurable ramp/random patterns. See the Parameter Editor's Rendering tab help for full details.

Animation

Animation requires Animating to be enabled (configured in the Run tab of the Parameter Editor). Three animation modes are available per sprite:

Draw background once

When Draw Background Once is enabled in the Run tab, the background is painted only on frame 1. Subsequent frames are drawn on top, accumulating a layered trail. With a semi-transparent overlay colour, older frames fade slowly — a fading-trail effect. Disable for clean per-frame animation.

Capture & Output

All output files go into the renders/ sub-directory of the current project, or a custom path if set in the Run tab.

Still capture

Press S in the Loom window, or click Save Still in the Parameter Editor's Run tab. The current frame is saved as:

renders/stills/loom_still_<N>.png

where N is an auto-incrementing integer. The PNG is always at full resolution (width × quality pixels), regardless of any on-screen scaling.

Animation recording

Press V (or Save Animation in the Run tab) to start recording. Each subsequent animation frame is saved as:

renders/animations/loom_frame_<NNNNN>.png

Press V again (or click the button) to stop. The resulting sequence of PNGs can be assembled into video with an external tool such as ffmpeg:

ffmpeg -framerate 30 -i loom_frame_%05d.png -c:v libx264 -pix_fmt yuv420p output.mp4

Resolution

Output resolution = Canvas Width × Quality Multiple by Canvas Height × Quality Multiple. A 1080×1080 canvas at quality 4 → 4320×4320 px PNG. At quality 8 → 8640×8640 px. Stroke widths are scaled proportionally if Scale Image is enabled in the Parameter Editor.

Reload Workflow

The reload mechanism lets you update any parameter in the editor and see the result without restarting Loom:

  1. Make changes in the Parameter Editor (any tab).
  2. Press Cmd+R (or click Reload) in the editor. The editor auto-saves all config files first.
  3. Loom detects the reload signal (a sentinel file .reload placed in the project directory) and re-reads all XML files.
  4. The canvas reinitialises and redraws. The SBT process is not restarted — reload is near-instant (typically under 1 second).

You can also press F5 directly in the Loom window to trigger a reload without going through the editor.

For the fastest iteration cycle: keep the Parameter Editor and Loom windows side-by-side. Edit a parameter, press Cmd+R, and see the result immediately. No SBT recompilation happens on reload — only the config files are re-read.

Parameter Editor Integration

The Loom Parameter Editor drives the Loom engine via saved XML files and sentinel files written to the project directory. There is no live socket connection — all communication is file-based.

Editor actionWhat happens
Run Loom (Cmd+L)Editor saves all config files, then runs sbt run in a child process.
Reload (Cmd+R)Editor saves all config files, then writes .reload sentinel file to project dir. Loom picks it up within one draw cycle.
Save Still (Run tab)Writes .capture_still sentinel file. Loom captures the current frame on the next draw cycle.
Save Animation (Run tab)Writes .capture_video sentinel file. Loom starts/stops recording.
Stop (Cmd+H)Terminates the SBT child process.
The sentinel file mechanism means the editor and engine can also be run as completely separate processes. You can edit XML config files by hand and press F5 in the Loom window to reload without the editor running at all.

Config Files Reference

All config files live in the project's configuration/ sub-directory. All are read by the Loom engine on startup and reload.

FileContents
global_config.xmlCanvas dimensions, background colour/image, quality multiple, animating flag, draw-background-once flag, 3D flag.
polygons.xmlPolygon set references (file paths relative to polygonSets/) and regular polygon definitions. Supports subdivision and morph targets.
curves.xmlOpen curve set references (file paths relative to curveSets/). No subdivision; rendered directly.
points.xmlPoint set references (file paths relative to pointSets/). No subdivision; rendered directly.
subdivision.xmlAll subdivision params sets. Each set contains one or more SubdivisionParams stages with type, transforms, and control point settings.
shapes.xmlShape configurations. Each shape has a source type (polygonSet, openCurveSet, pointSet) and a reference to the geometry source and (for polygons) the subdivision params set.
sprites.xmlSprite definitions: shape reference, renderer set reference, position, size, rotation, and animation configuration (jitter, keyframe, morph).
rendering.xmlRenderer set library. Each set contains one or more named renderers with mode, colours, stroke width, and dynamic change parameters.
